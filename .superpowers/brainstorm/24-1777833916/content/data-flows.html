Data flows — the two journeys

Section 3 of the design. These are the only two user-initiated flows that hit external services in v1.

Flow A

Add Account — first time a clan member adds an alt

1
User →
MainWindow
Clicks "Add Account" button.
2
MainWindow →
CookieCapture
CaptureAsync() — opens modal. v1: fresh WebView2 user-data folder per call.
3
CookieCapture →
WebView2
Navigates to https://www.roblox.com/login.
4
User →
Roblox web
Logs in normally (username + password + 2FA if enabled). The login form is the real Roblox one inside the embedded browser — we never see the password.
5
Roblox web →
WebView2
On successful auth, sets .ROBLOSECURITY cookie on roblox.com domain.
6
CookieCapture →
WebView2.CookieManager
After redirect to /home, polls GetCookiesAsync(".ROBLOSECURITY"). When found, closes modal.
User closes modal first → returns Cancelled. Login fails → returns Failed("Login was unsuccessful").
7
CookieCapture →
IRobloxApi
GetUserProfileAsync(cookie) → { userId, username, displayName }.
8
CookieCapture →
MainWindow
Returns Success(cookie, userId, username).
9
MainWindow →
AccountStore
AddAsync(displayName, cookie). Store also fetches avatar URL via IRobloxApi.
10
AccountStore →
DPAPI
v1: serialize accounts list to JSON, encrypt entire blob with ProtectedData.Protect() (CurrentUser scope), write atomically to %LOCALAPPDATA%\ROROROblox\accounts.dat.
11
MainWindow →
User
New account row appears in list with avatar + display name.
Flow B

Launch As — clan member starts Roblox on a saved account

1
User →
MainWindow
Clicks "Launch as Sarah_Alt".
2
MainWindow →
AccountStore
RetrieveCookieAsync(accountId). Decrypts via DPAPI, returns plaintext cookie (held briefly in memory).
DPAPI decrypt fails (profile changed/restored) → "Can't read saved accounts. Re-add them."
3
MainWindow →
RobloxLauncher
LaunchAsync(cookie).
4
RobloxLauncher →
auth.roblox.com
POST /v1/authentication-ticket with cookie. First call returns 403 + X-CSRF-TOKEN header. Replay POST with token. Receives RBX-Authentication-Ticket header.
401 → cookie expired → return CookieExpired → MainWindow offers re-auth via WebView2.
5
RobloxLauncher →
RobloxLauncher
Constructs URI:
roblox-player:1+launchmode:play+gameinfo:<ticket>+launchtime:<now-ms>+placelauncherurl:<...>+browsertrackerid:<rand>+robloxLocale:en_us+gameLocale:en_us
6
RobloxLauncher →
Windows shell
Process.Start(new ProcessStartInfo(uri) { UseShellExecute = true }). OS hands off to registered roblox-player:// handler.
Roblox not installed → shell can't resolve protocol → return Failed("Roblox does not appear to be installed.") with link to roblox.com/download.
7
Roblox launcher →
RobloxPlayerBeta.exe
Launcher reads ticket, spawns the actual game client. Mutex is already held by ROROROblox, so client doesn't see itself as the singleton — multi-instance works.
8
RobloxLauncher →
AccountStore
TouchLastLaunchedAsync(accountId) — updates LastLaunchedAt for sort/recency UX.
9
RobloxLauncher →
MainWindow
Returns Started(pid). UI shows "Launched ✓" toast for 2s.
Invariants worth calling out
  • — Plaintext cookies live in memory only during a single launch operation. Never written to disk unencrypted, never logged.
  • — Password never touches our process — login happens inside Roblox's own page, served by Roblox into WebView2.
  • — No HTTP request from us masquerades as a browser; we identify with a clear UA string (ROROROblox/<version>) so Roblox can rate-limit / fingerprint us cleanly if they want to.
  • — The only Roblox endpoints we call are documented public ones used by Bloxstrap and prior account managers. No undocumented endpoints, no scraping.
