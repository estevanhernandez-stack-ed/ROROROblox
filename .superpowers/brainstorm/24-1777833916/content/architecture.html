ROROROblox — Architecture (v1.1)

Single WPF .NET 10 process, tray-resident, opens main window on demand. Components below talk through clear interfaces; each is independently testable.

Process
ROROROblox.exe
MutexHolder
Acquires ROBLOX_singletonEvent at startup, holds for app lifetime. Toggle from tray.
TrayService
Hardcodet NotifyIcon. Right-click menu: Multi-Instance ON/OFF · Open · Quit.
MainWindow
WPF + WPF-UI. Hidden by default. Account list, Add Account, Launch As.
AccountStore
CRUD over saved accounts. Encrypts cookies with DPAPI before persistence.
CookieCapture
Modal WebView2 → roblox.com/login → reads .ROBLOSECURITY from cookie store.
RobloxLauncher
Cookie → auth ticket via Roblox API → spawns RobloxPlayerLauncher.exe -t <ticket>.
Windows OS
  • → Named Mutex (kernel)
  • → DPAPI (per-user vault)
  • → WebView2 runtime
Roblox
  • → auth.roblox.com (login + ticket exchange)
  • → users.roblox.com (avatar fetch)
  • → RobloxPlayerLauncher.exe
Filesystem
  • %LOCALAPPDATA%\ROROROblox\
  • └ accounts.dat (DPAPI)
  • └ logs/
  • └ webview2-data/
Why this shape
  • — Each component has one purpose and a clear interface — testable in isolation
  • — No background services, no admin rights, no kernel drivers — runs as plain user
  • — Roblox-side calls go through the same auth flow Bloxstrap and RAM use — no novel attack surface
  • — DPAPI-per-user means an account on machine A can't be opened on machine B (intentional v1 limit)
⚠ One thing to verify before we ship

The exact auth-ticket exchange endpoint Roblox uses today (auth.roblox.com/v1/authentication-ticket + X-CSRF-TOKEN dance) needs verification against current Roblox web behavior. Bloxstrap and RAM forks both use this pattern, but Roblox has tweaked it before. We'll prove it works in a 30-min spike before locking the launcher implementation.
