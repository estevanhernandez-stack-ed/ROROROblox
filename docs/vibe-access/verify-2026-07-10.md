## Method

Agnostic path (the v0.1 engine has no .NET adapter; its fetch() verifier can't speak
gRPC-over-named-pipe). A hand-built C# driver, compiled from the manifest + the published
`plugin_contract.proto` only, drove all 17 rpcs cold over the named pipe. Session-auth calls
ran as an unregistered, unconsented plugin — a held gate (`PermissionDenied` /
`Unauthenticated` / `FailedPrecondition`) is the pass; an OK response without consent is the
fail. Results were fed through the engine's own `stampManifest` + `renderVerifyReport` for
artifact parity.

Host under test: the production `PluginHostStartupService` + `CapabilityInterceptor` +
`PluginHostService` + `PluginUITranslator` classes, hosted headless on the production pipe
name (`rororo-plugin-host`) by a console app that references the WPF project. The GUI shell
was bypassed because its startup leftover-Roblox modal blocks the pipe on a box with live
Roblox clients, and that modal's clean-up action would kill the user's real sessions. That
modal is itself tracked as a defect — see the agent-ops spec.

Note that the host binding at all is now a proof in its own right:
`PluginHostStartupService.StartAsync` calls `RpcMethodCapabilityMap.AssertExhaustive()` before
Kestrel binds, so a pipe that comes up is a pipe whose every rpc has a capability-map entry.

  permission, and the host refuses to start if the map misses a method.
