## Method

Agnostic path (the v0.1 engine has no .NET adapter; its fetch() verifier can't speak
gRPC-over-named-pipe). A hand-built C# driver, compiled from the manifest + the published
`plugin_contract.proto` only, drove all 16 rpcs cold over the named pipe. Session-auth calls
ran as an unregistered, unconsented plugin — a held gate (`PermissionDenied` /
`Unauthenticated` / `FailedPrecondition`) is the pass; an OK response without consent is the
fail. Results were fed through the engine's own `stampManifest` + `renderVerifyReport` for
artifact parity.

Host under test: the production `PluginHostStartupService` + `CapabilityInterceptor` +
`PluginHostService` + `PluginUITranslator` classes, hosted headless on the production pipe
name (`rororo-plugin-host`) by a console app that references the WPF project. The GUI shell
was bypassed because its startup leftover-Roblox modal blocks the pipe on a box with live
Roblox clients, and that modal's clean-up action would kill the user's real sessions.

### History

- Run **f7ebdbd1** (pre-fix): 14/16 pass. `update-ui` and `remove-ui` were **fail** —
  both accepted a bogus `UIHandle` from a caller that owned no UI (fail-open). Root cause:
  neither rpc is in `RpcMethodCapabilityMap`, so `CapabilityInterceptor` doesn't gate them,
  and the handlers didn't check handle ownership.
- Run **34cf3714** (this run, post-fix): 16/16. The service layer now refuses an
  unknown/foreign/headerless handle with `PermissionDenied` (one status for unknown vs
  foreign so callers can't probe which handle ids exist). Covered by 8 new unit tests on
  `PluginHostService` + a real-pipe harness test (`UpdateUI_RemoveUI_RefuseForeignHandle_HonorOwnedHandle`).
