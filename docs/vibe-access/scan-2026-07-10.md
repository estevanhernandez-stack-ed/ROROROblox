The agent surface is the v1.4 plugin host: a Kestrel gRPC server on the per-user named pipe
`\\.\pipe\rororo-plugin-host` (`PluginHostStartupService.DefaultPipeName`). There is no HTTP
route table; "routes" below are the rpcs of `rororo.plugin.v1.RoRoRoHost`. gRPC is always POST
over HTTP/2, so every method column reads POST.

## Auth vocabulary for this stack

- **none** ‚Äî callable on a fresh pipe connection with no handshake and no consent. The floor for
  every call is the pipe ACL: current Windows user only, no network exposure.
- **session** ‚Äî requires a handshaken plugin identity (`x-plugin-id` metadata) plus a
  user-granted capability in the DPAPI `ConsentStore`, enforced by `CapabilityInterceptor`
  against `RpcMethodCapabilityMap`. `UpdateUI`/`RemoveUI` carry no capability entry but are
  gated downstream by UI-handle ownership ‚Äî a handle only a handshaken plugin can hold ‚Äî so
  they are marked session, not none.
- **token** ‚Äî unused for this stack.

## Observations

1. **Unknown rpc names fail open.** `RpcMethodCapabilityMap.Required()` returns null for any
   method not in the map, and the interceptor treats null as ungated. The map covers all 16
   current rpcs, so there is no exposure today ‚Äî but a future contract addition that misses a
   map entry ships ungated by default. `IsKnown()` exists and is unused by the interceptor;
   flipping unknown‚Üídeny (or a startup assert that every service method has a map entry) would
   make this fail closed. Worth an issue before the next contract bump.
2. **`GetRunningAccounts` is a pre-handshake free read** exposing display names, Roblox UIDs,
   PIDs, and place identity to any same-user process that opens the pipe. Consistent with the
   documented threat model (per-user pipe, same-user processes are trusted), noted here so the
   trust boundary is explicit in the inventory.
3. No cookies, tickets, or credentials cross this surface anywhere in the contract ‚Äî launch is
   command-by-reference (`account_id`), and the DPAPI blob stays host-side.
