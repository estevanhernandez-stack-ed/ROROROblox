## TL;DR

Cut **v1.4.3.0** of ROROROblox. The release bundles plugin-lifecycle fixes
that have been sitting on `main` since v1.4.2.0, plus three small net-new
pieces of work, plus a manifest schema extension that future-proofs the
plugin contract. The work is half "ship what's already merged" and half
"land #66 + manifest fields", in that order.

Once v1.4.3.0 ships, the sibling plugin repo (`rororo-ur-task`) cuts a
**v0.2.2** that gates on `minHostVersion: "1.4.3"` so users on older stores
get a clean refusal instead of the silent-fail dance we just hit.

## Why now

Yesterday's user-side bug: fresh install of **Ur Task v0.2.1** on store
**RoRoRo v1.4.2.0** silently fails to start the plugin EXE — even after a
manual RoRoRo restart. Root cause: `PluginInstaller.InstallAsync` defaults
`AutostartEnabled = false` for new installs, and v1.4.2.0 doesn't have
`bee8c23` (install-starts-in-session). Net: the EXE is never spawned, ever,
until the user toggles autostart on AND restarts RoRoRo. This is exactly
task **#66** from the rc15 handoff — "Este hit the no-Launch gap multiple
times". It bit again.

The fix is half already done — `bee8c23` removes the symptom on the dev
build — and half not — the underlying "plugins can't bootstrap themselves
on a stale host" problem is structural and needs a manifest signal so
plugin authors can declare *"I need host X+"*.

## Net new work for v1.4.3.0

### 1. Per-plugin Launch button on PluginsWindow rows (closes #66)

A small button on the right edge of each plugin row that spawns the plugin
EXE on-demand. Belt-and-suspenders with `bee8c23` (install-starts-in-session)
for the case where autostart is intentionally off but the user wants to run
the plugin ad-hoc without restarting RoRoRo.

**Wiring:**

- UI: `PluginsWindow.xaml` — add `<Button Content="Launch" .../>` after the
  Autostart checkbox in the row template. Disabled state when plugin is
  already running. Use the brand-cyan accent (consistent with the other
  action buttons on that row).
- VM: `PluginsViewModel` — add `LaunchPluginCommand` (`RelayCommand`)
  bound on each `PluginRow`. Implementation calls
  `PluginProcessSupervisor.Start(InstalledPlugin)` — the public method
  `ec200c6` already added for exactly this.
- Running-state: `PluginRow.IsRunning` already exists (or the equivalent —
  check `PluginProcessSupervisor` for the running-set accessor). Bind the
  button's `IsEnabled` to `!IsRunning`. If a running-state observable
  doesn't exist yet, add one — keep it minimal.
- Status banner: on click, set `StatusBanner = "Launching {Name}..."` then
  flip to `"{Name} is running."` once the process is up; use the existing
  banner pattern from `ToggleAutostartAsync`.

**Test:** unit-test the command's enabled-state transitions; smoke-test
manually by installing Ur Task on a clean profile, NOT toggling autostart,
clicking Launch, watching the tray icon appear.

### 2. Manifest forward-flags (three optional fields, no schema bump)

**No `schemaVersion` bump.** Add three optional fields to `PluginManifest`,
`PluginInstaller`, and `manifest.json` schema. Optional in both directions:
old host + new manifest → ignore unknown fields, behave as before; new host
+ old manifest → fields absent, use sensible defaults.

`schemaVersion` stays at `1` — bumps are reserved for shape-breaking
changes, and adding nullable JSON fields is back-compat both ways.

**Field 1: `autostartDefault: "on" | "off"`** *(string enum)*

Manifest declares the *initial* autostart preference for a fresh install.
User can flip it later in the Plugins window; this just sets the default
when the consent record is created. Doesn't override an existing consent
record on re-install.

Wiring:
- `PluginManifest.cs`: add `string? AutostartDefault { get; init; }` with
  `[JsonPropertyName("autostartDefault")]`. Parse same way as `Icon` /
  `UpdateFeed` (optional, no required-field gate).
- `PluginInstaller.InstallAsync`: read `manifest.AutostartDefault`. If
  `"on"`, set `AutostartEnabled = true` in the new `ConsentRecord`.
  Default (null / `"off"` / unknown value) → `AutostartEnabled = false`
  (today's behavior).
- Reject unknown values at parse time (`"sometimes"` → throw
  `PluginManifestException`).

Why enum, not bool: leaves room for a future `"prompt"` value (ask user at
install time) without a second migration. Cheap insurance.

**Field 2: `minHostVersion: "1.4.3"`** *(string semver)*

Plugin declares the minimum RoRoRo version it requires. Host refuses
install if its own version is below this, with a clear error.

Wiring:
- `PluginManifest.cs`: add `string? MinHostVersion { get; init; }`.
- `PluginInstaller.InstallAsync`: after manifest parse, before fetching
  zip, compare `manifest.MinHostVersion` to the host's current version
  (`HostInfo.Version` or the equivalent assembly-version source). If
  `minHostVersion` > current → throw
  `PluginInstallerException($"This plugin requires RoRoRo {min} or newer. You're running {current}. Update RoRoRo and try again.")`.
- Use `System.Version` for the compare — semver pre-release tags should
  be tolerated (split on `-`, parse the numeric head). Don't over-engineer:
  4-part numeric `Major.Minor.Build.Revision` matches the host's own
  version scheme.
- If `minHostVersion` is null → no constraint, accept (today's behavior).

**Field 3: `entrypoint: "626labs.ur-task.exe"`** *(string, optional)*

Plugin declares its main EXE filename. Host honors it instead of guessing
from the install dir.

Wiring:
- `PluginManifest.cs`: add `string? Entrypoint { get; init; }`.
- `PluginProcessSupervisor` (wherever the "which EXE do I run" decision
  lives — likely `DefaultPluginProcessStarter` based on the v1.4.2.0 file
  list): prefer `manifest.Entrypoint` when set. Fall back to current
  guess-from-dir logic when null.
- Validate at install time: if `entrypoint` is set, confirm the file
  exists in the unpacked zip; throw `PluginInstallerException` if not.

## Order of operations

1. **Manifest forward-flags first** (Field 1 + Field 2 + Field 3,
   bottom-up: `PluginManifest` → `PluginInstaller` → `DefaultPluginProcessStarter`).
   Wire-level changes that #66 doesn't depend on.
2. **Per-plugin Launch button (#66) second**. UI work, builds on the
   already-merged `PluginProcessSupervisor.Start()`.
3. **Release plumbing third** — release notes, reviewer letter, version bump.
4. **Tag and submit.**
5. **Plugin-side follow-up** — once v1.4.3.0 ships from Partner Center
   (not just tagged — actually live on the store), open a worktree in
   `rororo-ur-task` and cut **v0.2.2**:
   - `manifest.json`: add `"autostartDefault": "on"`, `"minHostVersion": "1.4.3"`, `"entrypoint": "626labs.ur-task.exe"`.
   - No code changes.
   - Release notes: "Requires RoRoRo v1.4.3 or newer. Fresh installs now
     start automatically — no more autostart-toggle-and-restart."
   - Tag `v0.2.2`, push, CI builds and uploads release artifacts.
