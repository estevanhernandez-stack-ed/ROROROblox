## DIMENSION: architecture
SUMMARY: RoRoRo's architecture is in good shape for a fast-shipping solo product: Core genuinely has zero UI dependencies, the typed-HttpClient crash class is fenced with a dedicated guard test, and failure isolation (plugins, presence, startup checks) is consistently degrade-safe. The two real debts are (1) a cluster of cross-thread reads of the UI-owned ObservableCollection that can fault the presence poll loop silently — undoing the v1.5 ghost fix for the rest of the session — and (2) MainViewModel as a 2,349-line god object with 19 constructor dependencies that every new feature (tags, transport, update probe, defenders) has landed in. Neither is a today-crash, but the second is exactly where the next feature will hurt.
STRENGTHS:
  + Core layering is actually clean: src/ROROROblox.Core/ROROROblox.Core.csproj references only Logging.Abstractions, CsWin32, and ProtectedData — no PresentationFramework, no System.Windows usage anywhere in Core (verified by grep). The IMutexHolder/IAccountStore/IRobloxApi/IRobloxLauncher seam holds.
  + Typed-HttpClient single-ctor discipline held and is regression-guarded: RobloxApi, RobloxCompatChecker, and RobloxUpdateProbe each have exactly one public constructor, and src/ROROROblox.Tests/TypedHttpClientRegistrationTests.cs resolves all three through a real ServiceCollection — the past crash class can't silently return.
  + Disposal/lifetime story is deliberate: App.OnExit stops the plugin host off the dispatcher with a 2s bound (avoiding the documented sync-context deadlock), stops the presence loop, then disposes the provider; MutexHolder, RobloxProcessTracker, PresenceService, TrayService, and AccountStore all implement IDisposable with careful handle hygiene (RobloxProcessTracker disposes every Process handle it touches).
  + Failure-isolation discipline is consistent: every plugin-host wiring point in App.xaml.cs is try/caught so a corrupt consent.dat or broken plugins root degrades to 'no plugin support' instead of blocking launch; RunStartupChecksAsync isolates each startup check independently.
  + Decision-bearing comments that cite root causes, not just behavior — the UseCookies=false registration (App.xaml.cs:304-319) documents the exact multi-launch cookie-leak bug it prevents; the mutex-name resolve block documents the prior UI-thread deadlock it replaced. This is the rare codebase where comments carry the postmortems.
  + Pure, testable extraction where it matters: MainViewModel.ResolveLaunchTarget and AccountMatchesFilter/EvaluateFollow are static pure functions; LaunchEligibility computation works on value snapshots — the launch-precedence logic is unit-testable without standing up the VM.

## DIMENSION: correctness
SUMMARY: RoRoRo's runtime core is engineered with unusual care for a v1.x desktop app — degrade-safe remote-config ladders, serialized launch dispatch, atomic DPAPI writes, and well-reasoned defender/tracker timeout lockstep. The standout reliability defect is the v1.5 presence loop: one unhandled exception (account removed mid-poll, store hiccup, or a cross-thread ObservableCollection enumeration) permanently kills the poll loop with no recovery and no user-visible signal, silently reverting the app to the pre-v1.5 ghost-state behavior. A cluster of smaller threading races (Process.Exited subscription order, unsynchronized session-history dictionary, presence Stop/Dispose) and silent exception swallowing in RelayCommand round out the risk profile. Nothing found rises to data-loss-critical; the DPAPI-corrupt misclassification path comes closest but self-heals on reload.
STRENGTHS:
  + UseCookies=false on the RobloxApi HttpClient is documented at the registration site with the exact failure mode it prevents (App.xaml.cs:304-319) — the root cause of the every-alt-opens-as-first-user bug is locked in as a load-bearing comment.
  + Mutex-name resolution ladder (remote config -> last-known-good cache -> hardcoded default) in RobloxCompatChecker.ResolveMutexNameAsync is genuinely no-throw, gated by MutexHolder.IsValidName, and budgeted at 2s so first paint is never held hostage to the network.
  + RobloxLauncher serializes all launches through a SemaphoreSlim gate and the 5s InterLaunchThrottle + FIFO start-time claim in RobloxProcessTracker make multi-launch attribution deterministic, with a documented 3s StartTime grace for clock skew.
  + AccountStore is properly thread-safe (single SemaphoreSlim gate), does atomic temp-file+move writes, and avoids no-op DPAPI roundtrips on chatty toggles.
  + AppStorageDefender's single-active-defender takeover (Interlocked.Exchange + cancel previous) and consumed-signal wind-down show careful reasoning about the captcha cross-branding race, including an honestly documented known limitation for multilaunch-during-install.
  + Presence hold-last-known-on-empty design with the 429 limitation explicitly documented in the class header rather than papered over.
  + Global exception nets (Dispatcher/AppDomain/UnobservedTask) log before crashing instead of recovering blindly, and the deliberate decision not to mark dispatcher exceptions Handled is stated.
  + Tracker attach timeout extends to 120s only while RobloxPlayerInstaller.exe is running, held deliberately in lockstep with the defender's 120s identity hold — the two subsystems can't disagree.

## DIMENSION: security
SUMMARY: The crown jewel is handled well on its primary paths: accounts.dat is DPAPI-CurrentUser enveloped with atomic writes, the v1.6 export bundle is genuinely strong crypto (AES-256-GCM AEAD + PBKDF2-HMAC-SHA256 600k, fresh salt/nonce per export, fail-closed import, keys zeroed), the past multi-instance cookie-bleed bug is fixed and the UseCookies=false handler is correctly isolated to the only client that attaches .ROBLOSECURITY, logging redacts to path+presence-boolean, and the plugin gRPC surface never returns the cookie. The real gaps are residual/secondary: the most-recent WebView2 capture profile (a live logged-in Roblox session) is never wiped and survives account removal; the plugin x-plugin-id is self-asserted so the consent capability model is advisory, not enforceable; and PluginInstaller accepts http:// URLs with a same-origin SHA, so the integrity check provides no authenticity on a code-download path. None of these directly exfiltrate the cookie, so nothing rises to critical, but the WebView2 residual and the installer transport gap are worth fixing.
STRENGTHS:
  + Transport crypto (AccountTransportService.cs:24-166) is textbook: AES-256-GCM AEAD, PBKDF2-HMAC-SHA256 at 600k iterations (OWASP 2023 floor), a fresh 16-byte salt + 12-byte nonce per export so identical inputs produce different bytes, the iteration count written into the header for forward-compat, fail-closed import with a single deliberately-ambiguous error (no passphrase/padding oracle), and key+plaintext+passphrase bytes zeroed via CryptographicOperations.ZeroMemory. Export gated at >=12 chars (PassphraseStrength.cs:17).
  + UseCookies=false is correctly scoped (App.xaml.cs:316-319). RobloxApi is the only client that attaches .ROBLOSECURITY and it does so manually per-request (RobloxApi.cs:102,328,400,451,547,613); the other typed clients (compat, update probe, tray painter, plugin installer) never touch the cookie, so their default containers are harmless. The fix for the v1.2 multi-instance cookie-bleed bug is real and documented inline.
  + The plugin gRPC surface never exposes the cookie. The highest-privilege host capabilities (request-launch / launch-target) take only an accountId; the cookie is resolved host-side in the launch invoker (MainViewModelLaunchInvokerAdapter.cs) and never crosses the wire. No capability in the vocabulary (PluginCapability.cs) returns secret material.
  + Zip-slip is correctly blocked (PluginInstaller.cs:114-124) by canonicalizing each entry with Path.GetFullPath and checking StartsWith against a separator-terminated prefix; and there is no verify/extract TOCTOU because the SHA is computed over the in-memory byte[] that is then extracted from the same MemoryStream.
  + Logging discipline holds: CookieCaptureWindow logs only path + a cookiePresent boolean (CookieCaptureWindow.xaml.cs:112), DiagnosticsCollector exports only an account count never the store (DiagnosticsCollector.cs:51-55), and the diagnostics zip stream-redacts any .ROBLOSECURITY occurrence as defense-in-depth (DiagnosticsWindow.xaml.cs:247-256).
