# v1.9.0.0 testing round — plan

**Build:** [RORORO v1.9.0.0 (pre-release)](https://github.com/estevanhernandez-stack-ed/ROROROblox/releases/tag/v1.9.0.0) · **Catalog:** live on the latest release (all three Ur plugins) · **Store submission waits on this round.**

This is the Velopack build, which means the plugin marketplace is VISIBLE (it hides on Store/MSIX installs by design). Three feature tracks plus regression spot-checks. Source checklists: [`marketplace smoke`](../superpowers/smoke/2026-07-04-plugin-marketplace-smoke.md) (Phase B now unlocked), [`friends picker smoke`](../superpowers/smoke/2026-07-04-friends-from-main-follow-smoke.md).

## Setup

- [ ] Quit any running RoRoRo from the tray (single-instance guard — one at a time).
- [ ] Install [`RORORO-win-Setup.exe`](https://github.com/estevanhernandez-stack-ed/ROROROblox/releases/download/v1.9.0.0/RORORO-win-Setup.exe) from the **v1.9.0.0 tag** (not "latest" — latest is stable 1.8).
- [ ] Verify the About box says **1.9.0.0**.
- [ ] Heads-up on accounts: if your daily driver is the **Store** install, its saved accounts live in the Store package's sandbox and may not appear in this build. Use **export** on the Store build → **import** here, or re-add a couple of test accounts.
- [ ] Open the Plugins window: the **Available** section should list the Ur family. If it's empty, note it immediately — that's a catalog-fetch failure, not cosmetic.

## Track 1 — plugin marketplace (the headline)

- [ ] **Browse:** Available lists catalog plugins you don't have — name, publisher, version, description, Install button.
- [ ] **Install from catalog:** Install → consent sheet appears (same as paste-URL) → plugin lands in Installed. (SHA verification happens under the hood; a failure would error loudly.)
- [ ] **Update badge:** with an older ur-task installed (anything below 0.5.0), its row shows the magenta **Update available (x → y)** badge + Update button. Badge text must match reality: 0.5.0 is the current release.
- [ ] **Update:** click Update → new version installs. A plugin that was **running** relaunches on the new version; one that was **stopped stays stopped** (no surprise force-launch).
- [ ] **minHostVersion gate:** ur-afk needs host 1.8+ — on this build (1.9) it should install normally. (If any entry ever shows "Needs RoRoRo X+", Install must be disabled.)
- [ ] **No double-run:** Install/Update buttons disable while an install is in flight.
- [ ] **Offline:** disconnect network → reopen Plugins → Available is empty, no error, no hang; installed list + paste-URL still work.
- [ ] **Paste-URL still works** (the pre-marketplace path must not have regressed).

## Track 2 — friends-from-main picker

- [ ] **1. Default = main.** Open the Friends picker on an **alt** row → it shows the **main's** friends; header names the main; hint reads "Follow one to launch [Alt] into their server."
- [ ] **2. Switch sources.** "View [Alt]'s friends" → list switches to the alt's own friends; header updates; hint hides.
- [ ] **3. Launch identity.** From the main's list, Follow an in-game joinable friend → the account that launches is the **alt**, not the main.
- [ ] **4. Main's own row:** no switch button; behaves as before.
- [ ] **5. No main set:** picker on any row shows that row's own friends, no switch button.
- [ ] **6. Rapid-toggle:** hammer the switch button — the list must never show a mixed set of both accounts' friends; switch/Refresh disable during load.
- [ ] **7/8. Expired sessions:** expired-account messages name the account whose fetch actually failed; an expired **main** still lets you switch to the alt's own list.

By design (not bugs): following into a friends-only server where the alt isn't the target's friend lands at home — best-effort, spec'd.

## Track 3 — reauth-tag regression (the fix that took two tries)

- [ ] Get an account into a needs-relogin state, re-login it → the tag clears and **stays cleared** through the next few background status cycles (the old race re-flagged it within ~30s).
- [ ] Re-login outcomes all surface: success, cancel mid-flow, and Roblox asking 2FA mid-flow — each should say what happened, never silently vanish.

## Track 4 — 1.8.1 riders (spot-checks, 5 minutes)

- [ ] **Close Roblox for me** (from the blocked modal or the banner) waits for Roblox to actually exit before re-acquiring — no immediate "still locked" flicker on a slow-closing client.
- [ ] A plugin installed but never consented shows the **consent sheet on first Launch**.
- [ ] `%LOCALAPPDATA%\ROROROblox\logs\` — current log is readable, plugin start/stop lines present, no flood.

## Reporting

Anything off: [open an issue](https://github.com/estevanhernandez-stack-ed/ROROROblox/issues/new) or drop it in Discord with the **track + check + what you saw**, plus the log from `%LOCALAPPDATA%\ROROROblox\logs\`. Screenshots welcome.

## Exit criteria → Store submission

- Track 1 fully green (the marketplace is the release), Track 2 checks 1-6 green, Track 3 green.
- Any Track-1 or Track-3 failure blocks; Track-2 edge cases (7/8) and Track-4 items triage case-by-case.
- Fixes found here land as **1.9.1.0** and the Store gets that instead (playbook bump rule).
